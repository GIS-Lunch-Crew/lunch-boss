# CONTEXT.md — lunch-boss

Living design document for the lunch-boss Forge app. Captures every decision made
during initial design discussion (July 2026), the reasoning behind each, and the
implementation roadmap. Update this file as decisions change.

Companion to `AGENTS.md` (Atlassian-generated agent instructions — see
[Platform Constraints](#platform-constraints) for how we treat it).

---

## 1. Product Goal

A lunch picker app for the Atlassian Marketplace.

**v1 scope — single user.** Each user gets their own setup where they can:

- Add restaurants to their personal pool (saving them for later selection)
- Remove restaurants from their pool
- Select a restaurant and "place" an order — placing = logging it as a record
  in the app (no real ordering integration)
- Pick a random restaurant from their pool
- Clear a loaded (in-progress, not yet placed) order
- See all previous orders
- See orders filtered by date

**Deferred to later versions:** group orders, single orders composing group
orders, multiple orders per day handling, per-user preferences.

The only *shared* data across users in v1 is the restaurant list itself — any
user's saved pool draws from a common per-site restaurant table.

## 2. Platform Constraints

These come from the Forge platform and will break the app if violated:

- **Forge is serverless.** There is no `server.js`, no HTTP server, no routers,
  no middleware. Atlassian hosts our backend as a function (see `manifest.yml`
  `function` module). `src/resolvers/index.js` registers named RPC endpoints.
- **Resolvers are named functions.** `resolver.define('key', fn)` registers a
  function; the frontend calls it with `invoke('key', payload)` from
  `@forge/bridge`. The resolver key *is* the route.
- **Resolver request shape:** data arrives on `req.payload` (NOT `req.body` —
  there is no HTTP request object). `req.context` carries invocation context,
  most importantly `req.context.accountId` — the authenticated Atlassian
  account ID of the caller.
- **Frontend must use UI Kit components from `@forge/react` only.** No raw
  HTML elements, no third-party React component libraries. (Full component
  list in `AGENTS.md`.)
- **Forge storage (SQL, KV, custom entities) has NO client-side API.** Every
  read/write goes through a backend resolver. This is why this app is
  resolver-heavy despite `AGENTS.md` suggesting frontend-direct API calls
  where possible — that advice applies to *product* REST APIs, not storage.
- **Forge SQL is scoped per installation** (per Confluence site). Restaurants
  are therefore shared among users of one site, which matches the v1 intent.

**About `AGENTS.md`:** generated by `forge create`; it is Atlassian's
instruction file for AI coding agents. Its *platform constraints* (UI Kit
component list, storage rules, CLI/deploy workflow) are real and must be
followed. Its *style preferences* are defaults we may override — and we
override one: it suggests vanilla JavaScript; **we use TypeScript** (decision
below).

## 3. Decisions & Rationale

### 3.1 No sign in / sign up

Users are already authenticated by Atlassian before they can reach the app.
`req.context.accountId` identifies the caller on every invoke. "Sign up"
collapses to "first time we see this accountId." No auth code exists in this
app.

### 3.2 No PII persistence

We do **not** store user name or email in our database. Atlassian
privacy/data-residency expectations (GDPR erasure, profile changes) make
persisted PII a liability. Instead:

- Store only `account_id` strings in our tables.
- Fetch the current user's display name **once per page load** from the
  frontend (`requestConfluence('/wiki/rest/api/user/current')` via
  `@forge/bridge`) and hold it in React state for the session. This requires
  the `read:confluence-user` scope in `manifest.yml` (add when built).
- For rendering other users, UI Kit's `User` component takes an `accountId`
  and renders name/avatar itself with no API call of ours.

Persist ≠ cache: session-level caching in frontend memory is fine; durable
storage of PII is what we avoid.

### 3.3 No `users` table (yet)

`account_id` from `req.context` is the user key everywhere — it joins and
filters fine without a `users` table backing it. A `users` table is added only
when we have per-user data that isn't a relation (preferences, defaults).

### 3.4 TypeScript

Overriding `AGENTS.md`'s vanilla-JS preference. Forge supports TS natively,
and typed entities flowing resolver → service → repository pay for themselves
given we're designing real relational schemas. Decided while the codebase was
~40 lines, so conversion cost was near zero.

### 3.5 Surface: Confluence global page

Swapping the scaffolded `macro` module for **`confluence:globalPage`** — the
app gets an entry in Confluence's Apps menu and renders a full page.

Why: a macro must be inserted into a page by someone, which is the wrong fit
for a personal tool you "pull up." There is no Forge module for a
popover-available-anywhere; global page is the closest clean fit. A macro can
return later as a secondary embed surface.

Jira-vs-Confluence for Marketplace is deliberately deferred: services,
storage, and validation code port across products almost untouched — only the
manifest module and frontend shell are product-specific.

### 3.6 Storage: Forge SQL

Chosen over KV Storage / custom entities because the data is relational
(shared restaurants, per-user saved lists, orders with date-range queries).
Forge SQL is a provisioned MySQL-compatible database accessed via `@forge/sql`
from backend resolvers only.

### 3.7 Validation: zod at the resolver boundary

Every resolver validates `req.payload` against a zod schema **before**
anything touches a service or SQL. Example: an order attempt with no selected
restaurant is rejected by `createOrderSchema.parse()` in the resolver — a
clean error returns to the frontend and no SQL executes.

### 3.8 Migrations define schema; constraints are the backstop

Migrations are `CREATE TABLE` / `ALTER TABLE` statements run **once at deploy
time** via the `@forge/sql` migration runner — they never see requests and are
not a validation mechanism. Their constraints (`NOT NULL`, composite `UNIQUE`)
are the last-resort backstop if a bug slips past zod and the services.

The Express-style "models folder" bundles three jobs that are three separate
things here:

| Express models job     | Here                                  |
|------------------------|---------------------------------------|
| Schema definition      | migrations (`storage/migrations.ts`)  |
| Request validation     | zod schemas (`validation/`)           |
| Row↔object translation | inline in repositories (see 3.9)      |

### 3.9 Mappers: skipped for now, inline in repositories

Mappers translate DB-dialect rows (`snake_case`, string dates/decimals) into
app-shaped objects (`camelCase`, proper types). Real but small problem at our
table count — repositories handle it inline (`SELECT account_id AS accountId`
gets most of the way). Extract a `mappers/` module only when conversions start
repeating or getting hairy (nested joins, computed fields). Refactor when the
pain shows up, not scaffolding on day one.

### 3.10 Restaurant identity & lifecycle

- **Duplicate = exact match on ALL THREE of (name, phone, address).** Two
  same-name restaurants at different addresses are distinct; two different
  restaurants sharing an address/phone are distinct. Only the full triple
  matching marks a duplicate.
- **"Exact" means normalized:** the service compares trim-to-lowercase values
  while storing what the user typed. (Prevents `"Joe's Pizza"` vs
  `"joe's pizza "` becoming two restaurants.)
- **Enforced in two layers:**
  - Service layer (primary): normalized duplicate lookup on create/edit. On
    duplicate during *create*, don't error — link the existing restaurant to
    the user's pool instead (turns the failure case into the desired
    feature). On *edit*, run the same check against every row except the one
    being edited — but unlike create, a collision on edit is **rejected with
    an error** (silently merging two restaurants would be surprising).
  - Migration layer (backstop): composite `UNIQUE (name, phone, address)`.
- **NULL gotcha:** MySQL unique indexes treat `NULL != NULL`, so optional
  fields are stored as `NOT NULL DEFAULT ''` and the app treats `''` as "not
  provided" — keeps the composite constraint airtight.
- **Soft delete:** `deleted_at` column; reads filter `WHERE deleted_at IS
  NULL`; order history keeps its restaurant reference intact.
- **Resurrect-on-readd:** a soft-deleted restaurant still occupies its unique
  triple. If a create matches a soft-deleted row, the service clears
  `deleted_at` instead of inserting.
- **Anyone in the org can edit any restaurant; edits are global.** No
  ownership checks in the update path.

### 3.11 Join table name: `user_saved_restaurants`

Explicit over clever. Each row reads as exactly what it is: "a restaurant this
user has saved." Rejected alternatives: `pool_entries` (too opaque),
`user_restaurant_selections`/`_picks` (collide with the app's existing
"select a restaurant" and "pick random" vocabulary — the action that creates a
row here is *saving*). Note it's not a textbook junction table (the user side
is an external identifier, not a table) but the SQL pattern is identical.

### 3.12 Order lifecycle: three stages

An order moves through three stages with different persistence needs:

1. **Selecting** — frontend state only. User picks a restaurant manually or
   randomizes; can re-pick/re-randomize freely. Nothing is persisted; a page
   reload loses the selection **by design** (re-picking is one click).
2. **Submitted** — exactly one per user, persisted in
   `user_current_submission`. The restaurant is **locked** (edits to it are
   rejected by validation — re-selecting requires clearing first);
   `total`/`notes` remain editable. Visible from any device. From here the
   user may **edit**, **clear** (delete the row), or **place**.
3. **Placed** — the service inserts an `orders` row and deletes the
   submission row. **Placed orders are immutable** — no edit/delete resolvers
   in v1; all editing happens during the submitted stage. The user can then
   start a new selection.

While a submission exists, the user cannot submit another order — enforced by
the database itself via `PRIMARY KEY (account_id)` on the submission table,
which also resolves the two-devices-submitting-at-once race for free (the
second insert fails cleanly; frontend shows the existing submission).

**Submit invariant:** submitting an order to a restaurant resurrects it if
it was soft-deleted and ensures it's in the caller's pool (idempotent save).
This makes re-orders of deleted restaurants work with zero side effects
until submit, and gracefully covers the race where a restaurant is deleted
while someone has it selected. Placed-order history always shows restaurant
names regardless of deletion (its JOIN ignores `deleted_at`).

**Re-order (frontend-only staging):** a history row's Re-order button stages
the selection stage with that order's restaurant (`SelectionTarget`:
id + name) and its items/total/notes (`OrderPrefill`) — no backend call and
no restaurant lookup happens until submit. Prefill clears on re-pick random
or cancel. Unavailable while a submission exists.

Rejected alternative: a `status` column on `orders` (`submitted` → `placed`).
MySQL lacks partial unique indexes, so one-submitted-per-user would need a
generated-column trick, and it mixes a mutable in-flight record into an
otherwise immutable history table.

## 4. Data Model

### `restaurants` — shared per site

| column       | type / notes                                   |
|--------------|------------------------------------------------|
| `id`         | PK, auto-increment                             |
| `name`       | `NOT NULL`                                     |
| `phone`      | `NOT NULL DEFAULT ''` (`''` = not provided)    |
| `address`    | `NOT NULL DEFAULT ''` (`''` = not provided)    |
| `website`    | optional                                       |
| `menu_url`   | optional                                       |
| `created_by` | `account_id` of creator (attribution only — no ownership rights) |
| `deleted_at` | nullable timestamp — soft delete               |
|              | `UNIQUE (name, phone, address)` — backstop     |

### `user_saved_restaurants` — a user's pool

| column          | type / notes                                |
|-----------------|---------------------------------------------|
| `account_id`    | Atlassian account ID (external identifier)  |
| `restaurant_id` | FK → `restaurants.id`                       |
|                 | `UNIQUE (account_id, restaurant_id)`        |

### `user_current_submission` — the one in-flight order per user

| column          | type / notes                                        |
|-----------------|------------------------------------------------------|
| `account_id`    | **PRIMARY KEY** — enforces one submission per user   |
| `restaurant_id` | FK → `restaurants.id`, `NOT NULL`, immutable after submit |
| `items`         | free-text order contents, editable while submitted ("items" because ORDER is a reserved SQL keyword) |
| `total`         | optional decimal, editable while submitted           |
| `notes`         | optional, editable while submitted                   |
| `submitted_at`  | timestamp                                            |

Row lifecycle: created by `submitOrder`, deleted by `clearSubmission` or by
`placeOrder` (which converts it into an `orders` row). See §3.12.

### `orders` — placed-order log (immutable history)

| column          | type / notes                                |
|-----------------|---------------------------------------------|
| `id`            | PK, auto-increment                          |
| `account_id`    | who placed it                               |
| `restaurant_id` | FK → `restaurants.id`                       |
| `items`         | free-text order contents                    |
| `ordered_at`    | timestamp                                   |
| `total`         | optional decimal                            |
| `notes`         | optional                                    |

Queries like "all my orders" / "my orders on date X" are
`WHERE account_id = ?` plus an `ordered_at` filter.

## 5. Data Flow

```
UI Kit frontend (src/frontend/)
  │  invoke('resolverKey', payload)          — all app data
  │  requestConfluence(...) via bridge       — current-user profile only
  ▼
Resolver (src/resolvers/)                    — thin: zod-validate payload,
  │                                            read accountId from context,
  │                                            call service, return JSON
  ▼
Service (src/services/)                      — business logic: duplicate
  │                                            checks, resurrect logic,
  │                                            create-vs-link decisions
  ▼
Repository (src/storage/)                    — the ONLY layer touching
  │                                            @forge/sql; inline row mapping
  ▼
Forge SQL (per-installation MySQL)
```

- Order lifecycle (§3.12): only the **selecting** stage is frontend-only.
  Once submitted, the in-flight order lives in `user_current_submission`
  server-side (survives reloads, visible from any device) until cleared or
  placed.
- "Pick random restaurant" operates on the already-fetched saved list in the
  frontend (`Math.random`) — no resolver needed. Revisit if pick logic grows
  (weighting, history-aware).

## 6. Resolvers (the API surface)

Grouped by domain; keys are the invoke strings. All resolvers read the caller
from `req.context.accountId` — the frontend never sends its own identity.

### Restaurants / pool (`src/resolvers/restaurants.ts`)

| key                    | payload                        | behavior |
|------------------------|--------------------------------|----------|
| `getSavedRestaurants`  | —                              | List caller's pool (joined restaurant details, excludes soft-deleted) |
| `addRestaurant`        | `{ name, phone?, address?, website?, menuUrl? }` | Normalized duplicate check → create new / link existing / resurrect soft-deleted; always ends with restaurant in caller's pool |
| `removeSavedRestaurant`| `{ restaurantId }`             | Remove from caller's pool only (restaurant row untouched) |
| `updateRestaurant`     | `{ restaurantId, ...fields }`  | Global edit, any user; duplicate check excluding self — collision rejected with error. UI: the add form doubles as the edit form (Edit button on a row pre-fills it; heading/buttons switch to edit mode) |
| `deleteRestaurant`     | `{ restaurantId }`             | Soft delete (sets `deleted_at`), global |

### Orders (`src/resolvers/orders.ts`)

| key                    | payload                            | behavior |
|------------------------|------------------------------------|----------|
| `getCurrentSubmission` | —                                  | Returns `{ submission: CurrentSubmission \| null }` (wrapped — never bare null, see §7) — page load uses this to decide which UI state to show |
| `submitOrder`          | `{ restaurantId, items?, total?, notes? }` | Create the caller's submission; rejects if one already exists (PK backstop); resurrects a soft-deleted restaurant + ensures pool membership (§3.12 submit invariant) |
| `updateSubmission`     | `{ items?, total?, notes? }`       | Edit submission details (overwrite semantics); `restaurantId` not accepted (restaurant is locked — clear to re-select) |
| `clearSubmission`      | —                                  | Delete the caller's submission; nothing written to history |
| `placeOrder`           | —                                  | Read caller's submission → insert `orders` row → delete submission |
| `getOrders`            | `{ from?, to? }`                   | Caller's placed orders, optional date range, newest first |

(Expect this list to grow; add rows here as resolvers are added.)

## 7. Project Structure

```
src/
  index.ts                 # re-exports the resolver handler (manifest entry)
  frontend/
    index.tsx              # UI Kit app (global page)
  resolvers/
    index.ts               # creates Resolver, registers all definitions
    restaurants.ts         # thin handlers: validate → service call
    orders.ts
  services/                # business logic, pure where possible
    restaurantService.ts
    orderService.ts
  storage/                 # ONLY layer that imports @forge/sql
    migrations.ts          # schema DDL + migration runner wiring
    restaurantRepository.ts
    savedRestaurantRepository.ts
    orderRepository.ts
  validation/              # zod schemas used at resolver boundaries
    restaurantSchemas.ts
    orderSchemas.ts
  types/                   # shared entity types (Restaurant, Order, ...)
    index.ts
```

Layer rules:
- Resolvers never contain business logic or SQL.
- Services never import `@forge/sql` or `@forge/resolver`.
- Repositories never make decisions (no duplicate checks — they expose
  `findByNormalizedIdentity`, the *service* decides what to do with it).

Engineering rules learned in development:
- **Cross-entity refresh:** when a backend action has side effects beyond
  the entity the handler is about, the frontend handler must refetch every
  affected collection. Current instances: submit → pool (may resurrect or
  pool-link a restaurant), place → history, delete → pool. Check this
  explicitly whenever adding a new cross-entity behavior — a missed refetch
  looks like the action "didn't work" until a page reload.
- **Never return bare `null` from a resolver.** The bridge delivers it to
  the frontend as an empty object `{}` (truthy), which is indistinguishable
  from real data. Wrap nullable responses in an object — see
  `CurrentSubmissionResult` in `src/types/index.ts`.

## 9. Team Development Workflow

Decided July 2026, kept deliberately simple while the team is small; revisit
at Marketplace prep (see "Consolidation" below).

**Current model — one app per developer:**
- Each dev registers their own Forge app. The **canonical/team app ID** is
  what lives in the committed `manifest.yml`; each dev swaps in their own ID
  locally and hides the diff with
  `git update-index --skip-worktree manifest.yml`. Personal IDs are noted in
  the gitignored `.env` — documentation only, nothing reads that file.
- **Committing manifest changes** (the only time the dance matters):
  `--no-skip-worktree` → set the canonical ID back → commit/PR → restore
  your ID → re-apply skip-worktree. Forgetting the last step means your next
  deploy lands on the team app instead of yours.
- **Merging deploys nothing.** GitHub and Forge are disconnected; code
  reaches an app only when someone with deploy rights runs `forge deploy`
  against that app ID. After a merge (or, at this early stage, acceptably
  before), someone deploys the canonical app and runs
  `forge install --upgrade` on the team site, then runs migrations.

**Key platform facts driving the workflow:**
- Environments live in Atlassian's cloud, not on any machine. A deployed +
  installed environment is usable by every user of that site in the browser.
- Forge SQL is scoped **per installation** (app + environment + site):
  separate apps or environments = separate databases. Shared test data only
  happens inside one installed environment.
- `forge tunnel` serves local *code*, but manifest-level state (modules,
  scopes) and applied migrations are server-side environment state — after a
  manifest change the environment needs a deploy + upgrade before tunnelling.

**Reviewing a teammate's PR — three modes for three purposes:**
1. *Does it work / UX check:* no commands. Open the team site in the browser
   and use the author's installed environment — you see their deployed code
   AND their database state.
2. *Is the code right:* the GitHub diff. No Forge involvement.
3. *Runtime debugging:* pull the branch and deploy it to **your own**
   environment (+ `install --upgrade` if the manifest changed, + run
   migrations). Your own WIP is safe in your branch — environments are
   disposable deploy targets, not workspaces. Convention: **never deploy to
   an environment/app named after someone else.**

**Convention:** each dev installs their dev environment/app on the team
space so teammates can browser-test PRs (mode 1) without pulling anything.

**Versioning:** Forge assigns app versions server-side at deploy (module and
scope changes auto-bump the major version); see the developer console.
`package.json`'s name/version are template leftovers Forge never reads.

**Consolidation (before Marketplace):** the listing points at a single app
ID, so the team consolidates then: one app, teammates added as collaborators
in the developer console, each dev deploying to their own custom environment
(`forge deploy -e <name>`). That deletes the skip-worktree dance and the
ID-swapping entirely. Do it as its own deliberate migration, not mid-feature.
